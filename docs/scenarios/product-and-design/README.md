# 产品、需求与设计协作

适合产品经理、设计师、创业者、业务负责人和研发团队。

## P1 内容计划

| 主题 | 状态 | 重点内容 | 计划路径 |
|---|---|---|---|
| 用户反馈转需求列表 | 🔵 调研类来源可复用 | 原话、问题、频次、影响、证据 | `feedback-to-requirements.md` |
| PRD 初稿 | ⚪ 待研究 | 背景、目标、用户、流程、验收、非目标 | `prd-draft.md` |
| 需求拆分和优先级 | ⚪ 待研究 | 用户价值、成本、风险、依赖、证据 | `requirements-prioritization.md` |
| 用户故事和验收标准 | ⚪ 待研究 | 角色、目标、价值、边界和异常 | `user-stories.md` |
| 竞品功能矩阵 | 🔵 竞品调研来源可复用 | 功能、目标用户、价格、证据和日期 | `feature-comparison.md` |
| 产品路线图草稿 | ⚪ 待研究 | 目标、阶段、依赖、风险、不确定性 | `product-roadmap.md` |
| 设计需求 Brief | ⚪ 待研究 | 目标、受众、尺寸、内容、品牌、交付物 | `design-brief.md` |
| UX 文案和错误提示 | ⚪ 待研究 | 场景、用户动作、可理解性、一致性 | `ux-writing.md` |
| 原型评审问题清单 | ⚪ 待研究 | 主流程、异常、权限、数据、可访问性 | `prototype-review.md` |
| 发布说明和帮助文档 | 🔵 文档能力可复用 | 用户变化、升级步骤、已知问题 | `release-help-docs.md` |

## 强制标准

- 不得把模型推测当成真实用户需求；
- 用户画像、痛点和优先级必须能追溯到数据、访谈或业务事实；
- PRD 中明确非目标、异常流程和验收标准；
- 设计图和品牌素材必须确认版权；
- 需求优先级由负责人决定，AI 只能提供结构化参考；
- 涉及隐私、权限、支付和删除的流程必须单独评审。

## 每篇教程需要的配图

- 用户问题到需求的证据链；
- PRD 标准结构；
- 用户流程和异常分支图；
- 优先级矩阵；
- 需求、设计、研发和验收的协作流程。
