# 用 WorkBuddy 写周报：从工作记录到可核对的汇报

> 验证状态：B 级来源核对。官方文档把周报列为典型办公场景；本文模板尚未完成本项目的完整人工实测。

好的周报不是把每天做过的事全部复制一遍，而是说明本周完成了什么、产生了什么结果、遇到什么问题、下周准备怎么推进。

## 开始前准备

可以把这些材料放入工作目录：

- 每日工作记录；
- 已确认的会议纪要；
- 项目任务列表；
- 本周数据；
- 上一周周报；
- 下周计划和截止时间。

不要直接把整个聊天软件导出记录交给 Agent。先筛选与周报有关的内容，并移除无关个人信息。

## 可直接复制的任务描述

```text
请根据 input/ 中的工作记录、会议纪要和数据，生成本周周报草稿。

要求：
1. 按“本周目标、完成事项、结果数据、问题风险、下周计划、需要支持”组织；
2. 合并重复事项，不要按日期简单堆砌；
3. 每项尽量写出结果和影响，不只写动作；
4. 数字必须来自输入材料；
5. 没有数据支持的成果不要夸大；
6. 未完成事项说明原因和下一步；
7. 不确定的信息标记“待确认”；
8. 输出一版 500 字以内简版和一版详细版；
9. 保存到 output/weekly-report.md；
10. 不要发送或发布。
```

## 把动作改成结果

可以让 WorkBuddy 帮你识别哪些句子只有动作，没有结果：

```text
请检查周报中的每一项：
- 如果只有“做了什么”，请提示我补充结果；
- 如果包含无法验证的夸张词，请标记；
- 不要自行编造数字或影响。
```

例如：

```text
动作：优化了登录流程。
结果：优化登录流程，将主要步骤从 5 步减少到 3 步；上线后数据待观察。
```

第二句仍需要人工确认数字是否真实。

## 给不同对象生成不同版本

```text
请基于同一份事实生成两个版本：
1. 给直属领导：突出结果、风险和需要协调的事项；
2. 给项目组：突出进度、依赖和下周行动项。

不要改变事实和数字，只调整信息顺序和表达重点。
```

## 检查清单

- 是否遗漏本周最重要的结果；
- 是否把计划写成已经完成；
- 是否出现输入材料中没有的数字；
- 是否把团队成果全部写成个人成果；
- 风险是否有明确影响和下一步；
- 下周计划是否包含负责人或截止时间；
- 是否保留了不应公开的客户或内部信息。

## 周报自动化之前先手动验证

周报非常适合后续设置每周自动任务，但至少先手动运行两到三次，确认：

- 输入来源稳定；
- 输出结构符合团队习惯；
- 不会读取无关目录；
- 不会自动发送未经确认的内容；
- 历史周报不会被覆盖。
