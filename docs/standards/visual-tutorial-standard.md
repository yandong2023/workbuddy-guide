# WorkBuddy 图文教程标准

本标准用于把仓库中的所有教程逐步升级到“普通用户照着做也能成功”的质量。

## 一、每篇教程的最低结构

正式教程必须包含：

1. 这篇教程解决什么问题；
2. 适合谁、不适合谁；
3. 预计耗时和准备材料；
4. WorkBuddy 模式、模型、Skill、工作目录和权限建议；
5. 完整操作步骤；
6. 可直接复制的任务描述；
7. 每一步的成功标志；
8. 结果文件和验收方法；
9. 常见失败原因；
10. 撤销、恢复和重试方法；
11. 隐私、安全和数据去向；
12. 参考来源和更新记录。

## 二、配图最低要求

面向非技术用户的完整教程，原则上至少配 **2 张图**；P0 重点教程建议配 **4—8 张图**。

可以使用：

- 本项目实际操作时截取的 WorkBuddy 界面；
- 自己绘制的 SVG 流程图；
- 工作目录和文件结构示意；
- 输入与结果对比图；
- 风险、权限和恢复流程图；
- 示例数据和虚构任务的结果截图。

不得使用：

- 伪造的 WorkBuddy 界面截图；
- 未经授权搬运的第三方教程图片；
- 包含真实账号、邮箱、Token、客户姓名和业务数据的截图；
- 与当前版本明显不一致却未标注的旧截图。

## 三、推荐配图顺序

一篇标准教程建议采用：

```text
图 1：任务整体流程
图 2：工作目录或准备材料
图 3：WorkBuddy 关键操作界面
图 4：分批执行或处理逻辑
图 5：最终结果示例
图 6：失败恢复和安全边界
```

不需要为了数量强行配图。每张图都必须帮助用户理解一个难点。

## 四、实际截图规范

### 1. 截图前

- 创建专门的演示账号或测试任务；
- 使用虚构项目、虚构姓名和无敏感数据的样例文件；
- 隐藏菜单栏账号信息、目录中的真实用户名和通知内容；
- 确认当前 WorkBuddy 版本和操作系统。

### 2. 截图内容

- 保留足够上下文，让用户知道按钮在什么位置；
- 使用框线、编号或箭头标出关键区域；
- 一张图只说明一个主要动作；
- 截图下方写清楚用户应该点击什么、看到什么。

### 3. 文件命名

```text
assets/screenshots/<slug>/01-open-task.png
assets/screenshots/<slug>/02-select-folder.png
assets/screenshots/<slug>/03-review-plan.png
```

### 4. 截图元数据

教程中应写明：

```text
截图环境：macOS / Windows
WorkBuddy 版本：如可确认则填写
截图日期：YYYY-MM-DD
```

## 五、SVG 示意图规范

当无法获取可靠界面截图时，优先使用原创 SVG 解释流程，而不是伪造 UI。

SVG 应做到：

- 中文文字清晰；
- GitHub 页面可直接显示；
- 不依赖外部字体和脚本；
- 浅色背景，普通屏幕可读；
- 每张图有明确标题；
- 颜色只用于区分阶段和风险，不依赖颜色表达唯一信息；
- 文件中不嵌入外部图片、Cookie、追踪代码或远程链接。

推荐路径：

```text
assets/illustrations/<slug>/workflow.svg
assets/illustrations/<slug>/folder-structure.svg
assets/illustrations/<slug>/recovery.svg
```

## 六、任务描述标准

每段可复制指令至少说明：

- 输入文件或目录；
- 任务目标；
- 处理步骤；
- 输出格式和位置；
- 不允许做什么；
- 不确定信息如何处理；
- 完成后如何报告；
- 哪些操作需要人工确认。

示例骨架：

```text
请处理当前工作目录中的【输入】。

目标：
- 【最终结果】

步骤：
1. 先检查输入并报告结构；
2. 先生成计划，不要立即修改；
3. 经我确认后分批执行；
4. 每批保存结果和进度；
5. 最后合并并生成验证报告。

限制：
- 不修改、覆盖或删除原文件；
- 不访问工作目录之外的位置；
- 不确定的信息标记“待确认”；
- 涉及联网、安装、发送、发布或完全访问时先询问我。
```

## 七、每一步必须可验证

不要只写“等待 WorkBuddy 完成”。每一步都应写出：

- 应该看到什么；
- 应该生成哪个文件；
- 文件大致包含哪些内容；
- 失败时停止在哪一步；
- 用户如何判断可以继续。

示例：

```text
成功标志：output/summary.md 已生成，文件可以打开，并且标题、页数和来源文件名正确。
```

## 八、结果展示标准

结果图或示例不能冒充真实效果。应明确标注：

- `示例结果`：人工制作的格式样例；
- `实际实测`：本项目在明确环境中执行得到；
- `官方示意`：来自官方文档，并保留来源；
- `待实测`：根据来源和能力设计，尚未完整验证。

## 九、可信度等级

### A：已实测

需要：

- 完整执行；
- 真实结果或脱敏截图；
- `verified_at`；
- 测试系统和版本；
- 失败与修正记录；
- 可复现样例。

### B：来源核对

需要：

- 至少一个官方来源；
- 说明尚未完成本项目人工实测；
- 社区经验与官方事实分开；
- 版本敏感内容标注核对日期。

### C：自动草稿

只能位于 `drafts/` 或 Draft PR，不得进入正式教程目录。

## 十、版权和引用

- 第三方教程只用于研究、交叉核对和发现缺口；
- 不复制第三方的完整步骤、独特表达、图片和结果；
- 文章末尾列出实际参考过的来源；
- 引用短句只在必要时使用；
- 本项目的价值是重新实测、补全步骤、降低风险和提供原创配图。
